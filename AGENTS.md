\# AGENTS.md



本プロジェクトにおける AI 実装・ドキュメント更新の統制ルールを定義する。



\---



\# 1. 基本原則



・既存設計を最優先とする  

・最小変更で実装する  

・責務を越えた変更は禁止  

・不明点は推測で補完しない  



\---



\# 2. 正本ドキュメント（Single Source of Truth）



以下を正本とする。



\- ARCHITECTURE.md  

\- PROJECT\_CONTEXT.md  

\- TASK\_PLAN.md  



上記とコードの整合性を常に維持すること。



\---



\# 3. ドキュメント同期ルール



\## 3.1 更新トリガー



以下の場合のみドキュメント更新を検討する。



・責務が変わった  

・データ構造が変わった  

・公開仕様が変わった  

・タスク状態が変わった  



\---



\## 3.2 更新対象



変更内容に応じて対象を限定する。



| 変更内容 | 更新対象 |

|--------|--------|

| 設計変更 | ARCHITECTURE.md |

| 仕様変更 | README.md |

| タスク進行 | TASK\_PLAN.md |



\---



\## 3.3 更新原則



・必要最小限のみ変更  

・既存記述は極力維持  

・不要な再構成は禁止  



\---



\# 4. TASK\_PLAN ルール



\## 4.1 状態管理



\- TODO  

\- DOING  

\- DONE  



\---



\## 4.2 実行制約



・NEXT に指定されたタスクのみ実行可能  

・NEXT が無い場合は新規実装禁止  

・勝手にタスクを増やさない  



\---



\## 4.3 更新タイミング



・実装完了時  

・設計変更時  

・スコープ変更時  



\---



\# 5. 実装ルール



\## 5.1 禁止事項



・pipeline構造の変更  

・GUI責務の拡張  

・CLI仕様の変更  

・新規依存の追加  

・大規模リファクタリング  



\---



\## 5.2 許可事項



・最小限のバグ修正  

・明確な仕様追加  

・既存構造内での機能追加  



\---



\# 6. 例外ルール



・例外は ValueError を優先する  

・エラー文は明確にする  

・例外設計を統一する  



\---



\# 7. README 更新ルール（公開仕様）



\## 7.1 更新してよい条件



以下の場合のみ README 更新を行う。



・ユーザー操作に影響がある  

・新機能が追加された  

・設定方法が変わった  



\---



\## 7.2 出力義務



README 更新時は必ず以下を含める。



1\. 変更ファイル一覧  

2\. 変更内容の要約  

3\. 追加/変更コード全文  

4\. 確認結果  

5\. 影響範囲  

6\. 残課題  



\---



\## 7.3 禁止事項



・内部仕様を過度に書かない  

・推測や未確定情報を書かない  

・説明のための構造変更をしない  



\---



\# 8. 自動化・運用ルール



・自動実行（T17）は入口のみ  

・運用仕様は T24 で管理する  

・履歴保存は config で制御する  

・既存ファイル上書きは許容  



\---



\# 9. 拡張ポリシー



・機能追加は段階的に行う  

・1タスク = 1責務  

・スコープ外の拡張は禁止  



\---



\# 10. 実行統合ルール（追加）



開発効率向上のため、以下を許可する。



・単一タスクに限り「実装 → 実行 → 検証」を1プロンプトで実行してよい  

・ただし対象は TASK\_PLAN に存在するタスクのみとする  

・複数タスクの同時実行は禁止  

・変更範囲は従来ルール（最小変更・責務分離）を厳守する  



目的  

・開発の往復回数を削減する  

・Codexの実行能力を活用する  



\---



\# 11. 最重要ルール



・設計を壊さない  

・勝手に広げない  

・必ず整合性を取る  



\---



本ファイルは、AIによる暴走・過剰実装・設計崩壊を防ぐための統制ファイルである。
